Yes — people have built ways to automate ChatGPT through the browser (using Selenium, Playwright, Puppeteer, etc.) without using an API token. But there are some important caveats.

✅ What exists (real examples)
1. Selenium-based “unofficial APIs”

Projects like ChatGPTAutomation repo
 let you:

Log in via browser

Send prompts

Read responses

There are also repos like ChatGPT-unofficial-api-selenium
 that explicitly call this an “unofficial API.”

👉 How they work:

Launch browser (ChromeDriver)

Log into chat.openai.com

Find textarea → type prompt → press enter

Scrape response DOM

👉 Downsides:

Slow and brittle (UI changes break it) 
GitHub

Not suitable for production

Can violate terms of service depending on usage

2. Playwright-based headless automation

Projects like ChatGPT without login (Playwright)

Or general browser flows using Playwright

👉 Example flow:

Start headless browser

Navigate to ChatGPT

Inject prompt

Stream response

👉 Why Playwright is popular:

More reliable than Selenium in many cases

Handles modern web apps better (async, dynamic DOM) 
Multilogin

3. “Session reuse” automation (no API key)

Some setups:

Log in once manually

Reuse cookies/session

Script runs headless afterward

This is commonly described in automation guides like browser automation workflow example
.

4. LLM-controlled browser automation (newer trend)

Things like Playwright MCP

Let an AI control the browser itself

👉 Idea:

Instead of hardcoding selectors:

“type this in the chat box”

System translates to browser actions

This is becoming more common in 2025-era tooling 
codemify.com

⚠️ Big caveats (important)
1. Terms of service risk

Automating ChatGPT UI instead of using API is often against platform policies

Especially:

scraping responses

bypassing rate limits

running bots at scale

2. Fragility

These scripts break when:

UI changes

class names change

layout shifts

Unlike API → no stability guarantees

3. Performance limitations

Much slower than API:

full browser rendering

waits for typing/streaming

Even authors of these repos warn against production use 
GitHub

4. Anti-bot protections

You may hit:

captchas

login challenges

temporary blocks

🧠 Bottom line

Yes — it’s definitely been done.
Common stack:

Playwright / Selenium
+ logged-in session
+ DOM interaction (textarea + response scraping)

But:

✅ Good for: experiments, prototypes, personal tools

❌ Bad for: production systems, scaling, reliability

👍 If you’re deciding what to do

If you want reliable automation → use API

If you want cheap / hacky / no-key → browser automation works

If you want cutting-edge → look at LLM-controlled browser agents
